## 1. Политика версий SDK: дрейф typecheck `@1.2.0` vs runtime `@1` (major)

Примеры типопроверяются против lockfile-пиннинга `@bitrix24/b24jssdk@1.2.0`
(`.actualize/typecheck/package-lock.json`), а UMD-табы и пользователи используют
плавающий мажор-тег `@1`. Зелёный `tsc` на `1.2.0` не гарантирует валидность на
будущих `1.x`, которые подтянет `@1` → документация может молча разойтись с SDK,
и CI этого не заметит.

Предложение:
- периодический CI-job (`schedule` + `workflow_dispatch`): `validate.py` по всем
  `done`-файлам ledger против `@1 latest` (не против пиннинга), информационно;
- план бампа/отката: при breaking-change SDK — обновить `typecheck/package.json` +
  `package-lock.json`, прогнать `validate.py` по ledger, при провале — точечная
  ре-актуализация;
- (опц.) обсудить пиннинг UMD до минора `@1.2` для консистентности.

## 2. Batch-runner для массовой актуализации (~1547 файлов) (major)

Тулинг работает по одному файлу (`validate.py` — один путь, `record.py` — один
upsert). Для всего `api-reference/**` нужен оркестратор — отдельный PR:
- параллельный прогон `validate.py` по `remaining.py --list` (`xargs -P` / `parallel`);
- батч-режим `record.py` (N путей из stdin);
- ретраи; возобновляемость уже есть через идемпотентный ledger + `remaining.py`;
- управление таймаутами/лимитами при per-file прогоне агентом (фиксированные батчи).

Контракты `validate.py`/`record.py` не меняются — раннер строится поверх.
